# AstrBot E2B 云沙箱插件 (astrbot_plugin_e2b_sandbox)

![Platform](https://img.shields.io/badge/Platform-Windows%20%7C%20Linux%20%7C%20Docker-blue)
![License](https://img.shields.io/badge/License-AGPL--3.0-green)
![Python](https://img.shields.io/badge/Python-3.9+-blue)
![Status](https://img.shields.io/badge/Status-Active-brightgreen)

**使用 E2B 云沙箱安全执行 Python 代码，支持通过 LLM 自然语言调用。**

这是 AstrBot 的增强型代码执行工具，它将不安全的本地 Python 执行环境替换为隔离的云端沙箱。现在它不只是“能跑代码”，在文件处理、结果回传和自定义环境（Template）上已经有了一套非常顺滑的工作流。

## 💡 为什么用它？（核心优势）

- **💻 零服务器算力负担**：所有的代码计算、文件处理、画图都在 E2B 的云端服务器完成！**完全不占用你宿主机的 CPU 和内存**。哪怕你的 AstrBot 部署在 1核1G 的“乞丐版”小鸡上，照样能让大模型跑庞大的数据分析。
- **⚙️ 极简配置，开箱即用**：不用在宿主机折腾复杂的 Docker，也不用天天排查 Python 虚拟环境依赖冲突。装好插件，填个 API Key，立刻拥有完整的代码执行能力。
- **🛡️ 绝对安全隔离**：代码在云端执行，完全隔离于宿主机。再也不用怕模型发神经执行 `rm -rf /` 把你的服务器扬了。

## ⚠️ 重要前置要求（必读）

> [!IMPORTANT]
> 经实测，AstrBot 新版自带了“使用电脑能力”，请按以下说明配置以确保本插件完美运行：

### 🚫 不要开启全局 Sandbox
在 AstrBot 的管理面板中，进入 **“设置” -> “普通配置” -> “使用电脑能力”**，找到 **“运行环境”** 选项：
- 请将它设置为 `local` 或者 `none`。
- **千万不要设置为 `sandbox`**。
- **亲测：`local` 配合本 E2B 云沙箱插件可以顺利共存并完美运行**，功能不会冲突。

### 🌐 网络环境配置
E2B 服务端在海外。如果你的 AstrBot 部署在国内且网络不通，请在插件配置中填写 `proxy`（例：`http://127.0.0.1:10809`）。

## 🛠️ 安装与配置

### 1. 安装依赖
```bash
# 进入 AstrBot 的虚拟环境后执行
pip install e2b-code-interpreter>=2.2.2,<3.0.0
```

### 2. 获取 E2B API Key
访问 [E2B Dashboard](https://e2b.dev/) 登录并复制你的 API Key。注册即送 $100 永久抵扣金，按秒计费，对个人聊天场景来说基本等于永久免费。

### 3. 插件配置项

| 配置项 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---|---|---|
| `e2b_api_key` | 字符串 | 是 | 空 | E2B API Key |
| `timeout` | 整数 | 否 | 60 | 单次代码执行超时（秒） |
| `max_output_length` | 整数 | 否 | 2000 | 返回给 LLM 的最大文本长度 |
| `proxy` | 字符串 | 否 | 空 | E2B 请求代理地址 |
| `default_template` | 字符串 | 否 | 空 | 默认使用的 E2B Template ID |
| `user_whitelist` | 字符串列表 | 否 | 空 | 允许使用插件的用户白名单，填写用户 ID；留空表示禁用插件 |
| `max_return_file_size_mb` | 整数 | 否 | 5 | 允许缓存/发送的单个结果文件大小上限（MB） |
| `file_retention_hours` | 整数 | 否 | 24 | `data/plugin_data/astrbot_plugin_e2b_sandbox/exports/` 缓存文件保留时长（小时） |

## 🎮 使用示例

**1️⃣ 联网抓取**
* **用户**："看看 astrbot.app 首页的 title 是什么"
* **结果**：沙箱自带海外网络环境，自动联网抓取，LLM 根据返回结果直接告诉你网页标题。

**2️⃣ 绘图与文件分析**
* **用户**：(发了一个含有数据的 excel 文件) "帮我读取这个文件，并画一个饼图"
* **结果**：插件自动将你的文件塞进沙箱 -> 代码读取分析 -> 注入中文字体画图 -> 拦截图片并异步发回你的聊天窗口。

**3️⃣ 结果文件由模型自主发送**
* **用户**："把刚才那个表格里年龄大于 30 的人筛选出来，存成一个新的 excel 发给我。"
* **工作流**：
  1. 模型生成 `filtered_result.xlsx`
  2. 插件识别并缓存候选结果文件
  3. 模型调用 `e2b_sandbox_list_files` 查看候选列表
  4. 模型调用 `e2b_sandbox_send_file` 把真正想发的那个文件发给用户

## 💡 进阶：关于 E2B 模板 (Template) 的避坑指南

什么是模板？简单理解就是 E2B 里的 Docker 镜像。它可以让你提前装好笨重的依赖包，避免每次跑代码都要临时 `pip install`。

**怎么用最稳？**
1. **最省心的用法**：留空。如果你只是常规数据处理、画图、爬虫，直接用默认环境，什么都别管，最省事也最省钱。
2. **全局固定环境**：如果你自己（通过 E2B CLI/SDK）构建了一个包含了 `pandas`, `openpyxl`, `Pillow` 等常用库的模板，把它的 ID 填进配置里的 `default_template`，以后所有对话默认都从这个模板启动。
3. **按需指定环境**：如果你后面自己改成多工具版本，也可以额外扩展成“按次指定模板”的玩法；但当前 README 对应的默认版本里，主入口还是 `run_python_code` + `default_template`。

## 📝 注意事项与最佳实践

- **沙箱复用与资源释放**：当前会话的沙箱状态（变量、文件、依赖）会被保留，不再是每次清空的“一次性容器”。但请注意合理释放资源，任务结束后建议让模型暂停或销毁沙箱，避免长时间挂机空耗云端额度。
- **默认执行后自动暂停**：为控制成本，代码运行结束后沙箱默认会自动进入暂停状态。如果确实需要运行后台常驻任务，必须在提示词中明确要求模型“保持沙箱持续运行”。
- **结果文件按需精准发送**：沙箱生成的文件不会再被强制自动发到聊天中。它们会先存入本地缓存队列，由大模型根据你的要求，自主挑选并决定发送哪个最终结果给你，大大降低了错发率。
- **超大文件拦截保护**：为了防止撑爆宿主机的网络带宽和硬盘，超过设定体积上限的生成文件将被插件直接拦截并跳过。
- **图表绘制的稳妥写法**：当要求大模型进行数据可视化（画图）时，建议引导它使用“保存为图片文件”的方式（例如 `plt.savefig()`），而不是直接在代码中调用显示（`plt.show()`），以确保图片能顺利生成并回传。
- **屏蔽沙箱内部路径**：大模型有时会直接回复沙箱内的绝对物理路径（如 `/home/user/...`），这对本地聊天界面是无效的。建议提醒模型直接把处理好的文件发出来，而不是回复一条无法访问的路径。
