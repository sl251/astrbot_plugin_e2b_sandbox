**使用 E2B 云沙箱安全执行 Python 代码，支持通过 LLM 自然语言调用。**

这是 AstrBot 的增强型代码执行工具，它将不安全的本地 Python 执行环境替换为隔离的云端沙箱。现在它不只是“能跑代码”，在文件处理、结果回传和自定义环境（Template）上已经有了一套非常顺滑的工作流。

## 💡 为什么用它？（核心优势）

- **💻 零服务器算力负担**：所有的代码计算、文件处理、画图都在 E2B 的云端服务器完成！**完全不占用你宿主机的 CPU 和内存**。哪怕你的 AstrBot 部署在 1核1G 的“乞丐版”小鸡上，照样能让大模型跑庞大的数据分析。
- **⚙️ 极简配置，开箱即用**：不用在宿主机折腾复杂的 Docker，也不用天天排查 Python 虚拟环境依赖冲突。装好插件，填个 API Key，立刻拥有完整的代码执行能力。
- **🛡️ 绝对安全隔离**：代码在云端执行，完全隔离于宿主机。再也不用怕模型发神经执行 `rm -rf /` 把你的服务器扬了。

## ⚠️ 重要前置要求（必读）

> [!IMPORTANT]
> 经实测，AstrBot 新版自带了“使用电脑能力”，请按以下说明配置以确保本插件完美运行：

### 🚫 不要开启全局 Sandbox
在 AstrBot 的管理面板中，进入 **“设置” -> “普通配置” -> “使用电脑能力”**，找到 **“运行环境”** 选项：
- 请将它设置为 `local` 或者 `none`。
- **千万不要设置为 `sandbox`**。
- **亲测：`local` 配合本 E2B 云沙箱插件可以顺利共存并完美运行**，功能不会冲突。

### 🌐 网络环境配置
E2B 服务端在海外。如果你的 AstrBot 部署在国内且网络不通，请在插件配置中填写 `proxy`（例：`http://127.0.0.1:10809`）。

🎮 使用示例
1️⃣ 联网抓取
用户："看看 astrbot.app 首页的 title 是什么"
结果：沙箱自带海外网络环境，自动联网抓取，LLM 根据返回结果直接告诉你网页标题。
2️⃣ 绘图与文件分析
用户：(发了一个含有数据的 excel 文件) "帮我读取这个文件，并画一个饼图"
结果：插件自动将你的文件塞进沙箱 -> 代码读取分析 -> 注入中文字体画图 -> 拦截图片并异步发回你的聊天窗口。
3️⃣ 结果文件自动回传 (v1.1.0 当前行为)
用户："把刚才那个表格里年龄大于 30 的人筛选出来，存成一个新的 excel 发给我。"
工作流：
1. LLM 写代码过滤数据，生成 filtered_result.xlsx。
2. 插件捕捉到新文件，悄悄下载到本地。
3. 插件自动检查文件大小、完整性和候选优先级。
4. 插件自动把最可能是最终结果的那个文件发给你。

💡 进阶：关于 E2B 模板 (Template) 的避坑指南
什么是模板？简单理解就是 E2B 里的 Docker 镜像。它可以让你提前装好笨重的依赖包，避免每次跑代码都要临时 pip install。
怎么用最稳？
1. 最省心的用法：留空。如果你只是常规数据处理、画图、爬虫，直接用默认环境，什么都别管，最省事也最省钱。
2. 全局固定环境：如果你自己（通过 E2B CLI/SDK）构建了一个包含了 pandas, openpyxl, Pillow 等常用库的模板，把它的 ID 填进配置里的 default_template，以后所有对话默认都从这个模板启动。
3. 按需指定环境：如果你后面自己改成多工具版本，也可以额外扩展成“按次指定模板”的玩法；但当前 README 对应的默认版本里，主入口还是 `run_python_code` + `default_template`。
⚠️ 千万不要让大模型去“造”模板
技术上大模型能写 Dockerfile 去构建模板吗？能。但我强烈建议不要这么做。
目前的最佳实践是：人来维护模板，模型只负责“选择并使用”模板。 让模型自动构建模板极容易导致环境混乱和无意义的额度燃烧。

📝 注意事项
• 即用即焚：每次大模型调用工具都是一个全新的沙箱！不要第一句话让它定义变量，第二句话让它打印。如果需要连贯操作，请让大模型把逻辑写在一次代码调用里。
• 文件回传限制：超过 max_return_file_size_mb 的大文件会被插件直接抛弃，防止撑爆你的服务器带宽和硬盘。
• 画图不要用 plt.show()：虽然插件底层做了容错，但最稳的依然是让 LLM 调用 plt.savefig()。
